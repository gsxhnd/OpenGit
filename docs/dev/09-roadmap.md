# 9. 开发路线图

## 路线图原则

- **先闭环，再增强**: 优先打通 `打开仓库 -> 查看状态 -> 暂存 -> 提交 -> 同步远端` 的完整工作流。
- **先单仓库，再多仓库**: 单仓库体验稳定后，再扩展项目管理、分组、工作区和跨仓库操作。
- **先安全操作，再高风险操作**: rebase、reset、冲突解决等高风险能力必须具备预览、确认、错误恢复和清晰反馈。
- **先核心 Git，再生态集成**: Git 基础能力独立可用后，再接入 GitHub / GitLab / Gitea、AI、终端等扩展功能。
- **每阶段可交付**: 每个阶段都应产出可运行、可演示、可测试的版本。

## Phase 0: 项目基础与开发体系

> 目标: 建立可持续迭代的工程基础，降低后续功能开发成本。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| 工程结构梳理 | 明确 `app` / `git` / `ui` / `settings` 等模块边界；整理入口和依赖方向 | 0.5 周 |
| 应用启动流程 | 初始化窗口、主题、日志、配置目录、错误处理入口 | 0.5 周 |
| 配置存储 | 支持本地配置文件；保存窗口尺寸、最近仓库、主题偏好 | 0.5 周 |
| 日志与诊断 | 统一日志输出；记录 Git 命令、耗时、失败原因；提供诊断入口 | 0.5 周 |
| 错误模型 | 定义用户可读错误、技术错误、可重试错误；统一展示策略 | 0.5 周 |
| 基础测试 | 为 Git 后端、配置读写、关键状态转换建立测试样例 | 0.5 周 |

**交付物**: 可启动的应用骨架、稳定的配置与日志基础、可扩展的模块结构。

**验收标准**:

- 应用可在空配置下首次启动。
- 配置损坏时能回退到默认配置并提示用户。
- Git 操作失败时能显示明确错误信息，而不是静默失败或崩溃。

## Phase 1: 单仓库 MVP 闭环

> 目标: 实现可用的单项目 Git 客户端，覆盖日常提交与同步流程。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| 仓库打开 | 选择本地目录；校验 `.git`；记录最近仓库；显示当前仓库路径 | 0.5 周 |
| 仓库状态 | 展示 modified / added / deleted / renamed / untracked / conflicted 文件 | 0.5 周 |
| 文件暂存 | 支持单文件 stage / unstage；支持全部 stage / unstage；操作后自动刷新 | 0.5 周 |
| Commit 输入 | 提交标题、正文、字符提示、空消息校验、无暂存文件校验 | 0.5 周 |
| Commit 执行 | 创建提交；展示成功反馈；失败时保留输入内容 | 0.5 周 |
| Inline Diff | 展示新增、删除、上下文行；支持文件级 diff；处理二进制文件提示 | 1 周 |
| 分支基础 | 显示当前分支、分支列表；支持切换本地分支；有未提交变更时提示 | 0.5 周 |
| Commit 历史 | 分页加载提交列表；显示 hash、作者、时间、标题；支持选择查看详情 | 0.5 周 |
| 远端同步 | 支持 fetch / pull / push；展示进度、成功、失败和认证错误 | 0.5 周 |
| 状态栏 | 当前分支、变更数量、领先/落后、当前操作状态 | 0.5 周 |

**交付物**: 可打开本地仓库、查看变更、提交、查看历史、push / pull / fetch 的基础 Git GUI。

**验收标准**:

- 能完成一次真实仓库的修改、暂存、提交、推送流程。
- 文件状态和 diff 在 Git 操作后自动保持一致。
- 网络、认证、冲突等失败场景有明确反馈。

## Phase 2: 核心体验增强

> 目标: 提升单仓库使用体验，让基础功能接近日常可用标准。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| Diff 体验增强 | 支持 side-by-side diff；行号；文件切换；大文件降级提示 | 1 周 |
| Hunk 操作 | 支持按 hunk stage / unstage；支持丢弃单文件变更前确认 | 1 周 |
| 文件系统监听 | 监听仓库文件变化；防抖刷新；忽略 `.git` 内部噪音事件 | 0.5 周 |
| 通知系统 | Toast、错误详情、操作进度、可复制错误信息 | 0.5 周 |
| 分支操作增强 | 创建分支、删除分支、重命名分支、基于指定提交创建分支 | 0.5 周 |
| Remote 管理 | 列表、添加、编辑、删除 remote；校验 URL；显示默认 remote | 0.5 周 |
| Tag 管理 | tag 列表；创建 annotated / lightweight tag；删除和推送 tag | 0.5 周 |
| Stash 基础 | stash save / apply / pop / drop；显示 stash 列表与摘要 | 0.5 周 |
| 空状态与引导 | 无仓库、空历史、无变更、无远端时提供明确操作入口 | 0.5 周 |

**交付物**: 单仓库核心操作完善，具备高频 Git 工作流所需的操作反馈、diff 和分支管理能力。

**验收标准**:

- 用户无需命令行即可完成常见分支、tag、remote、stash 操作。
- 大文件、二进制文件、外部变更不会造成 UI 卡死。
- 所有破坏性操作都有二次确认或清晰的风险提示。

## Phase 3: 多项目与工作区

> 目标: 支持管理多个仓库，形成 OpenGit 的核心差异化工作区能力。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| 项目列表 | 添加、移除、重新打开仓库；显示仓库名称、路径、当前分支、变更数 | 0.5 周 |
| 项目持久化 | 保存项目列表、排序、最近打开时间、上次选中项目 | 0.5 周 |
| 项目分组 | 创建、重命名、删除分组；拖拽或菜单移动项目 | 0.5 周 |
| 多项目切换 | 支持侧边栏切换；保留每个项目的选中文件、滚动位置和视图状态 | 0.5 周 |
| Tab 工作区 | 支持多仓库 Tab；关闭、固定、恢复上次会话 | 0.5 周 |
| 批量状态刷新 | 并发刷新多个仓库状态；限制并发；失败项目单独提示 | 0.5 周 |
| 多项目概览 | 汇总展示变更数、未推送提交、远端落后、冲突状态 | 0.5 周 |
| 项目搜索 | 按名称、路径、分组搜索项目；支持键盘快速打开 | 0.5 周 |

**交付物**: 可稳定管理多个 Git 项目，支持项目分组、快速切换和跨项目状态概览。

**验收标准**:

- 50 个项目以内的列表与状态刷新保持可用性能。
- 某个项目刷新失败不会影响其他项目。
- 重启应用后能恢复项目列表和工作区状态。

## Phase 4: 历史、搜索与可视化

> 目标: 提供专业 Git 客户端所需的历史理解、定位和可视化能力。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| Commit 详情 | 展示完整 message、作者、提交时间、父提交、变更文件和 diff | 0.5 周 |
| Git Graph | 绘制 commit 拓扑；支持分支、merge、tag 标记；虚拟滚动 | 2 周 |
| 历史筛选 | 按分支、作者、日期、文件路径筛选 commit | 0.5 周 |
| Commit 搜索 | 按 message、hash、作者搜索；支持结果定位到历史列表 | 0.5 周 |
| 文件搜索 | 按文件名搜索工作区文件；支持打开 diff 或历史 | 0.5 周 |
| File History | 查看单文件提交历史；支持从历史中查看该文件 diff | 0.5 周 |
| Blame 视图 | 逐行显示提交、作者、时间；支持跳转到对应 commit | 0.5 周 |
| Reflog | 查看本地引用变动；支持复制 hash；危险恢复操作前确认 | 0.5 周 |

**交付物**: 可视化提交图、强搜索能力、文件级历史和 blame 分析能力。

**验收标准**:

- 大型仓库历史加载采用分页或虚拟列表，不阻塞主界面。
- 搜索结果能稳定跳转到对应 commit、文件或 diff。
- Graph 对 merge commit、tag、远端分支展示准确。

## Phase 5: 高级 Git 操作与冲突解决

> 目标: 覆盖复杂 Git 工作流，并通过清晰 UI 降低误操作风险。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| 冲突识别 | 标记冲突文件；区分 both modified / deleted by us / deleted by them 等状态 | 0.5 周 |
| 基础合并视图 | 展示 ours / theirs 区块；支持选择一侧、保留两侧、手动编辑 | 1 周 |
| 三方合并视图 | 展示 base / ours / theirs / result；保存结果并标记 resolved | 1 周 |
| Rebase 基础 | 展示待 rebase commit；支持 continue / abort / skip；冲突时进入解决流程 | 1 周 |
| 交互式 Rebase | reorder、squash、fixup、edit、drop；执行前展示计划预览 | 1 周 |
| Cherry-pick | 选择 commit 并应用；冲突处理；支持 abort / continue | 0.5 周 |
| Reset / Revert | 支持 soft / mixed / hard reset；revert commit；危险操作强确认 | 0.5 周 |
| Patch 操作 | 导出 patch；应用 patch；失败时展示 rejects 或错误详情 | 0.5 周 |

**交付物**: 支持冲突解决、rebase、cherry-pick、reset、revert 等高级 Git 操作的专业工作流。

**验收标准**:

- 高风险操作必须先展示影响范围和确认提示。
- 操作中断、失败或冲突时，用户能看到下一步可执行动作。
- 冲突解决保存后，Git 状态与 UI 状态保持一致。

## Phase 6: 效率工具与个性化

> 目标: 提升高频用户效率，支持键盘优先和个性化工作流。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| 快捷键系统 | 定义全局 action；默认快捷键；冲突检测；快捷键提示 | 0.5 周 |
| 命令面板 | `Ctrl+Shift+P` 打开；模糊搜索命令；支持最近命令和上下文命令 | 0.5 周 |
| 设置面板 | 通用设置、Git 设置、外观设置、快捷键设置、隐私设置 | 1 周 |
| 主题系统 | Light / Dark；跟随系统；主题变量；基础自定义能力 | 0.5 周 |
| AI Commit Message | 基于 staged diff 生成提交信息；多 Provider；Conventional Commits 模板 | 1 周 |
| Git Hooks 管理 | 查看 hooks；启用/禁用；编辑；执行失败时跳转到日志 | 0.5 周 |
| 内置终端 | 嵌入 PTY；默认进入仓库目录；支持多个终端会话 | 1.5 周 |
| 操作历史 | 记录最近 Git 操作、耗时、结果；支持重新查看错误详情 | 0.5 周 |

**交付物**: 支持命令面板、快捷键、设置、主题、AI 提交信息和内置终端的高效率客户端。

**验收标准**:

- 常用操作可通过键盘完成。
- AI 功能必须可关闭，且不会在未确认时上传敏感 diff。
- 设置修改后能即时生效或明确提示需要重启。

## Phase 7: Git 扩展能力

> 目标: 覆盖团队和大型仓库常见的 Git 扩展场景。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| Submodule 管理 | 列表、初始化、更新、同步、打开子模块仓库 | 0.5 周 |
| Worktree 管理 | 列表、创建、删除、打开 worktree；显示关联分支 | 0.5 周 |
| LFS 支持 | 检测 LFS；track / untrack；pull / push；prune；显示大文件状态 | 0.5 周 |
| Commit 签名 | GPG / SSH 签名配置检测；提交签名；历史中展示验证状态 | 0.5 周 |
| Sparse Checkout | 检测 sparse 模式；展示规则；支持基础启用和路径管理 | 0.5 周 |
| Ignore 管理 | 查看 `.gitignore`；从未跟踪文件添加 ignore 规则；支持全局 ignore 提示 | 0.5 周 |
| 大仓库优化 | 状态刷新缓存、历史分页、diff 限制、后台任务取消 | 1 周 |

**交付物**: 支持 submodule、worktree、LFS、签名、sparse checkout 等扩展 Git 能力。

**验收标准**:

- 扩展功能未启用时不干扰普通仓库体验。
- 大仓库操作有明确进度、取消和降级策略。
- worktree / submodule 操作不会破坏主项目列表状态。

## Phase 8: 平台集成与协作

> 目标: 接入主流代码托管平台，覆盖 PR / MR、CI 状态和协作流程。

| 任务 | 需求拆分 | 预估 |
|------|----------|------|
| 账号体系 | Token / OAuth 配置；多账号管理；凭据安全存储 | 1 周 |
| GitHub 集成 | PR 列表、创建 PR、查看 review 状态、CI 状态、打开网页 | 1.5 周 |
| GitLab 集成 | MR 列表、创建 MR、Pipeline 状态、打开网页 | 1 周 |
| Gitea 集成 | PR 列表、创建 PR、状态展示、打开网页 | 0.5 周 |
| Issue 关联 | 从分支名或 commit message 识别 issue；提供打开链接 | 0.5 周 |
| PR Diff | 查看 PR / MR 变更文件、提交列表、评论入口 | 1 周 |
| 通知聚合 | 展示待 review、CI 失败、远端变更等协作提醒 | 0.5 周 |

**交付物**: 可与 GitHub、GitLab、Gitea 协作流程打通的团队级 Git 客户端。

**验收标准**:

- 平台认证失败、权限不足、网络失败有可理解反馈。
- 未配置平台账号时，本地 Git 功能不受影响。
- 创建 PR / MR 前能清楚展示源分支、目标分支和提交范围。

> 目标: 达到可公开发布和长期维护的质量标准。

| 任务 | 需求拆分 | 预估 |

| Windows 适配 | 路径、换行、凭据、终端、文件监听、权限问题测试 | 1 周 |
| macOS 适配 | 菜单栏、快捷键、签名、公证、系统主题测试 | 1 周 |
| Linux 适配 | 桌面环境、字体、文件选择器、凭据存储、包格式测试 | 1 周 |
| 打包发布 | 生成安装包；版本号；更新日志；发布产物校验 | 1 周 |
| 自动更新 | 检查更新、下载、安装、失败回滚或手动安装提示 | 1 周 |
| 性能基准 | 冷启动、状态刷新、历史加载、diff 渲染、内存占用基准 | 0.5 周 |

**验收标准**:

- Windows、macOS、Linux 至少各完成一轮核心流程验证。
- 发布包可安装、启动、卸载，并能保留用户配置。
- 性能基准有记录，后续版本可对比回归。

## 优先级建议

| 优先级 | 范围 | 原因 |
|--------|------|------|
| P0 | Phase 0 - Phase 1 | 决定产品是否能完成最小 Git 工作流 |
| P1 | Phase 2 - Phase 3 | 决定是否具备日常使用价值和多项目差异化 |
| P2 | Phase 4 - Phase 5 | 决定是否达到专业 Git 客户端能力 |
| P3 | Phase 6 - Phase 8 | 决定效率、生态和团队协作体验 |
| P4 | Phase 9 | 决定公开发布质量和长期维护能力 |

## 版本切片建议

| 版本 | 覆盖阶段 | 发布目标 |
|------|----------|----------|
| `v0.1` | Phase 0 + Phase 1 核心闭环 | 内部自用，验证基础 Git 工作流 |
| `v0.2` | Phase 2 | 单仓库体验可用，支持高频操作 |
| `v0.3` | Phase 3 | 多项目管理可用，形成产品差异化 |
| `v0.4` | Phase 4 | 历史、搜索、Graph 能力可演示 |
| `v0.5` | Phase 5 | 高级 Git 操作进入可用状态 |
| `v0.6` | Phase 6 - Phase 7 | 效率工具和扩展 Git 能力完善 |
| `v0.7` | Phase 8 | 平台协作集成可用 |
| `v1.0` | Phase 9 | 跨平台发布与稳定性达标 |

## 风险与依赖

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| Git 操作复杂度高 | 容易出现状态不一致或误操作 | 建立统一 Git 后端抽象和操作状态机 |
| 大仓库性能压力 | 状态刷新、历史和 diff 可能卡顿 | 后台任务、分页、缓存、取消、虚拟列表 |
| 跨平台差异 | 文件监听、路径、凭据、终端差异明显 | 尽早建立三平台 smoke test 清单 |
| 高风险操作误用 | rebase / reset / clean 可能造成数据损失 | 预览影响范围、强确认、提供恢复指引 |
| 平台 API 差异 | GitHub / GitLab / Gitea 能力不完全一致 | 抽象平台能力模型，按能力降级展示 |
| AI 隐私问题 | diff 可能包含敏感信息 | 默认关闭、明确授权、可配置 Provider 和脱敏策略 |
